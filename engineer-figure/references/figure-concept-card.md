# Figure Concept Card

Use this before choosing chart types for any manuscript-facing figure, especially when
the user asks for a figure that should be creative, convincing, mechanism-oriented, or
more impressive than a basic metric plot.

## Purpose

Start from the paper argument, not from the available dataframe. A strong figure usually
combines one visual idea with several kinds of evidence. Metrics are often supporting
evidence, not the whole story.

## Required concept card

Create a short card in working notes before plotting:

```text
Core claim:
Reviewer question:
Visual job:
  compare metrics / explain mechanism / show process / reveal decision logic /
  show uncertainty / show system interaction / show spatial or trajectory evidence
Evidence roles:
  phenomenon:
  mechanism or process:
  quantitative validation:
  robustness/control:
Hero panel:
Supporting panels:
Reference route:
  engineering-reference-atlas ID / Nature archetype / demo pattern:
What not to use as the main view:
Risk if too conventional:
Risk if too decorative:
```

If the user only provides metrics, infer the likely mechanism or process behind the
metrics and decide whether a supporting schematic, decision map, process strip, or
uncertainty panel would make the claim more reviewable.

## Option generation

For broad or underspecified requests, generate 2-3 design options before coding:

- **Conservative option**: clear, familiar, low review risk.
- **Mechanism/process-led option**: one hero schematic, workflow, phase map, or decision
  landscape supported by small quantitative panels.
- **High-impact composite option**: more visually distinctive, but still scientifically
  defensible and not decorative.

If the user asked you to proceed directly, choose the best option and briefly state why.
Do not stop only to ask for a preference unless the choice would materially change the
scientific claim or require missing data.

## Evidence balance

Prefer a figure that answers at least two of these questions:

- What happens? (phenomenon or result)
- Why does it happen? (mechanism, causal path, algorithm logic, physical process)
- When or where does it happen? (phase, trajectory, spatial or temporal context)
- How reliable is it? (uncertainty, sensitivity, validation, robustness)
- Why should the reviewer trust it? (controls, source data, sample size, assumptions)

If all panels answer only "which metric is bigger", redesign unless the paper section
really is only a benchmark comparison.

## Reviewer-facing discipline

Creative does not mean ornamental. Reject visual ideas that:

- hide the measured quantities;
- require the reader to decode too many encodings;
- use novelty without adding a mechanistic or evidentiary layer;
- make the claim harder to audit;
- look like a dashboard rather than a manuscript figure.

The preferred creative move is to change the figure's **structure**: hero mechanism,
process sequence, decision landscape, uncertainty envelope, or system interaction map.
Do not rely on unusual colors or decorative shapes to create novelty.
